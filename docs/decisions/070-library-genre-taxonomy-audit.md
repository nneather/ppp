## Decided

- **The taxonomy is fundamentally sound — only one merge, not a redesign.** Before running the AI classifier against the full backlog, more than half the library (716 of 1,378 books) had no genre, which made several genres look dead or redundant (`Chinese Language Tools`, `German Language Tools` both had 0 live books). Once classified, every one of the 42 genres picked up real usage. Rejected hypotheses, with evidence: collapsing the Theology family (`Theology`/`Applied Theology`/`Biblical Theology`/`Historical Theology`/`Systematic Theology` all landed in the 12–38 book range post-classification — genuinely distinct sub-disciplines); collapsing `Church History`/`History`/`Historical Theology` (34/78/38 — distinct, well-populated); killing `General` as a duplicate of `Other` (title sampling shows `General` = general secular nonfiction/ideas books; `Other` = hobby/craft/travel/miscellany — different jobs, both real); killing `Language` as dead weight (title sampling shows it's linguistics/translation-as-topic, distinct from the instructional per-language tool buckets).
- **`Pastoral` → `Pastoral Ministry` merge, evidence-based.** Every sampled title in both (2 + 23 books) is the same subject (church shepherding/leadership). This closes an open item from decision 018, which flagged `Pastoral` (and `General`) as legacy Pass-1 import short-code labels (`PAS`/`GEN`) awaiting an owner rename decision. Decision 018's own suggested rename targets were `Pastoral Ministry` (confirmed correct) and `General Epistles` (**rejected** — real `General` content is secular nonfiction, nothing biblical; renaming to `General Epistles` would have been wrong).
- **`Biblical Reference`'s non-biblical entries are a known, deliberate Pass-1 import compromise, not a bug.** `migrationOverrides.ts` documents it explicitly: `BH: 'Biblical Reference', // Brockhaus is general reference; closest of the 12 closed genres` — Brockhaus (a general German encyclopedia/dictionary set) was mapped there because, at Pass-1 import time, the taxonomy only had 12 genres and `Biblical Reference` was the closest fit. The taxonomy has since grown to include a plain `Reference` genre. Left as a documented backlog item (see Open questions) rather than acted on unilaterally — re-tagging specific rows is an editorial judgment call, not a mechanical fix.
- **No genres added from the crosswalk itself** — nothing in the AI's full-backlog classification surfaced content that didn't fit an existing bucket. The 5 new genres below came from a direct owner request (canonical NT/OT division granularity), not from the data audit.
- **Decision questions were presented to the owner twice.** First pass was skipped entirely — proceeded on the single low-risk, strongly-evidenced merge (Pastoral) and left the rest open rather than assuming judgment calls. Second pass got real answers: keep `General Epistles`; extend the NT/OT split rather than flatten it (owner specified `Acts and Paul` for NT, `Pentateuch`/`Old Testament Historical Books`/`Psalms and Wisdom Literature`/`Prophets` for OT); run the `Biblical Reference` cleanup now. All three built same session (see Built above).

## Surprises (read these before the next session)

- **Pre-classification genre counts are not reliable signal for taxonomy decisions.** With 52% of the library unclassified, several genres looked dead (0 live books) that turned out to have real, confirmed demand once the backlog was actually run through the classifier (`Chinese Language Tools`, `German Language Tools`). Always run the AI pass (or at least sample the backlog) before judging a low-count genre as bloat.
- **`execute_sql` in Ask mode refuses even pure `SELECT` queries** — the MCP tool isn't scoped as read-only by statement content, so any live-data pull for an audit needs Agent mode, not just read-only file tools.
- **`execute_sql` also refuses `UPDATE`/`INSERT`/`DELETE` even in Agent mode** — `ERROR: 25006: cannot execute UPDATE in a read-only transaction`. The MCP server runs it in a read-only transaction unconditionally; the only way to write is through a migration (`apply_migration` / `supabase db push`). Good forcing function toward the migration-first workflow, but surprising the first time — don't waste a turn debugging the SQL when the tool itself is the blocker.
- **The full-backlog `--all --apply` run took ~20 minutes** (OL lookups throttled at 300ms/call + batched Anthropic calls, 725 candidates) — background it, don't block on it in the foreground.
- Two intermittent `HTTP 529 overloaded_error` responses from Anthropic mid-run; the script's per-batch handling absorbed them without failing the run (a few books just didn't get a genre proposal that batch — accounted for in the 725→648 gap along with genuine no-signal books).
