# 001 — Audit log UI

**Date:** 2026-04-25
**Module:** invoicing (cross-cutting)
**Tracker session:** Session 6

## Built

- `/settings/audit-log` page — newest-first paginated list of every `audit_log` row, owner-only.
- Filters: module (`all` / `invoicing` / `library`), `record_id` UUID search, `changed_by` UUID search. All filters are URL-driven so each filtered view is shareable and deep-linkable.
- Per-row collapsible diff. UPDATEs render a 3-column `field / before / after` table; INSERTs and DELETEs render a sorted key/value dump of the surviving payload.
- Revert button on UPDATEs only, gated by an explicit table whitelist. Wired through `ConfirmDialog`.
- Owner-only "Audit log" card surfaced on the `/settings` hub (the `profiles.role` check is now also returned from the hub `+page.server.ts`).
- Reusable `<AuditRow>` component with an exported `diffKeys()` helper.

## Decided (non-obvious)

- **Revert is UPDATE-only and table-whitelisted, not "anything `revertible = true`."** The trigger marks INSERTs as `revertible = true` because the data *is* in fact revertible (DELETE the row); but reverting an INSERT means deleting a record, which can violate FKs across modules. We chose the conservative cut: UI only reverts UPDATE rows, only on tables where whole-row replacement is safe. Whitelist: `clients`, `client_rates`, `time_entries`, `invoices`, `invoice_line_items`, `profiles`. Library tables are intentionally excluded — the polymorphic `(book_id OR essay_id)` CHECK and the `compute_verse_abs` trigger make whole-row patches risky; library will need its own revert UX.
- **The "DB decides data-revertibility, UI decides user-revertibility" split.** The trigger keeps a single semantic flag based on what would parse cleanly back into a row. The UI layers an additional safety policy on top. This means a row can show `revertible: true` in the diff inspector but no revert button; that's intentional and documented.
- **Strip list before patching.** `id`, `created_at`, `created_by`, `updated_at`, `verse_start_abs`, `verse_end_abs` are all removed from `old_data` before `supabase.from(table).update(patch)`. Rationale:
  - `id`/`created_at`/`created_by` are immutable identity.
  - `updated_at` is regenerated by `set_updated_at()` trigger.
  - `verse_*_abs` are computed by `compute_verse_abs()`. (Currently moot because library tables aren't in the whitelist, but the strip set is module-agnostic so it's future-safe when library revert lands.)
- **Invoice status guard.** Even when an UPDATE row is whitelisted-revertible, we re-fetch the current invoice and refuse the revert if current status is `sent`/`paid` and the would-be patched `status` differs. This prevents reverting "back through" a status transition that should require an explicit business action (un-send, re-issue).
- **The revert itself fires the audit trigger and creates a new audit row.** This is desirable, not a bug — the log shows both the original change *and* the revert as distinct entries with `changed_by = current owner`.
- **Diff comparison uses `JSON.stringify` per key.** Cheap, deterministic, and good enough for primitives + small JSONB payloads we have today. `updated_at` is in a noise-key skip list so every UPDATE doesn't trivially show that one field.
- **Native `<select>` for the module filter, not the shadcn Select primitive.** The filter form is a plain GET form (URL-driven). Native select round-trips for free; the shadcn Select would have required a hidden input + JS onchange handler with no UX win.
- **Underscore-prefixed exports.** SvelteKit rejects non-standard runtime exports from `+page.server.ts` (`PAGE_SIZE`, `INVOICING_TABLES`, `LIBRARY_TABLES`, `REVERTIBLE_TABLES`). They're prefixed with `_` so the constants can be co-located with the action that uses them and re-exported into the component. Type-only exports (`AuditLogRow`, `AuditOperation`) don't trip the validator.

## New components / patterns added

- `src/lib/components/audit-row.svelte` — diff-rendering audit list row. Exports `diffKeys()` and `formatValue()` helpers from a `<script context="module">` block. Add to [`.cursor/rules/components.mdc`](../../.cursor/rules/components.mdc).
- Pattern: **module-scoping via table whitelist**. `_INVOICING_TABLES` and `_LIBRARY_TABLES` constants live in `+page.server.ts`; library will extend `_LIBRARY_TABLES` as new tables ship. Filter chips → `.in('table_name', tables)` is the entire mechanism.
- Pattern: **owner gate as friendly load error, not redirect**. RLS already blocks viewer reads of `audit_log`; the UI adds a server-side `profiles.role` check so viewers see a clear message instead of an empty list.

## Open questions surfaced

- Does library revert ever need to land? If yes, the strip list already covers `verse_*_abs`, but the polymorphic CHECK on `book_id` / `essay_id` means a partial UPDATE could violate the constraint. Library decision deferred — see [`docs/POS_Library_Session_0.md`](../POS_Library_Session_0.md) decision-point list.
- Do we want a "revert all changes since X" bulk action? Not in MVP; the per-row revert with a new audit trail is sufficient for August.
- Pagination is currently `range(offset, offset + 50)`. At 10k+ rows that's slow (`OFFSET` scans). Acceptable for MVP — the changed_at index gives reasonable performance through ~100k rows. Future: keyset pagination on `(changed_at, id)`.

## Surprises (read these before the next session)

1. **SvelteKit `+page.server.ts` rejects non-standard runtime exports** with a clear error message (`Invalid export 'PAGE_SIZE'... or anything with a '_' prefix`). Easy fix once you know — prefix with `_` and import from `./+page.server` into the component. Do NOT try to extract the constants into a separate `_constants.ts` because the action also needs them, and importing across files just to satisfy the linter is noise.
2. **`profiles.full_name` is nullable**, so the changed-by display falls back to email. The audit trigger uses `auth.uid()` for `changed_by`, so seed-data inserts that ran without an authenticated session show as `system` (i.e. `changed_by IS NULL`). Useful signal — anything labelled `system` in the log is non-user (seed, migration, or service-role) traffic.
3. **`ConfirmDialog` doesn't accept its own form**. The pattern (mirroring `settings/invoicing`) is: render a hidden `<form>` outside the dialog, set its hidden input from JS when the user clicks "Confirm", then `formEl.requestSubmit()`. The dialog is purely chrome.
4. **The audit log viewer is stop-energy for "what did I do last week?"-style debugging**. Watching the diff column is much faster than re-reading commit messages. Worth adding to the Library Session 0 acceptance: a smoke test that the same UI surfaces library writes once those tables get used.
